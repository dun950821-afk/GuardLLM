# 检测数据持久化方案

## 概述

修复检测流程的数据记录缺失问题，创建统一的检测会话记录服务，确保检测工作台、多模型评测等场景的检测数据正确写入数据库，让检测看板和历史记录展示真实数据。平台：web。依赖：Supabase 数据库。

## 技术方案

| 维度 | 选择 | 理由 |
|------|------|------|
| 记录服务 | 创建统一 Recorder 模块 | 复用性强，字段映射集中管理 |
| 写入时机 | 检测完成后立即记录 | 保证数据完整性，便于追溯 |
| 调用位置 | 前端页面调用检测后 | 可控制是否记录，区分测试场景 |

## 功能模块

### 1. 检测记录服务 (`src/lib/detection/recorder.ts`)
- 提供 `recordDetectionSession()` 方法
- 参数：用户输入、检测结果、模型信息、延迟等
- 写入 `detection_sessions` 和 `detection_records` 表
- 返回会话 ID

### 2. 检测工作台数据流
```
用户输入 → 输入检测 → 模型对话 → 输出检测 → 记录会话
```
- 完整记录输入/输出检测结果
- 记录脱敏/改写处理结果
- 记录模型响应

### 3. 多模型评测数据流
```
测试用例 → 检测 → 记录会话
```
- 每个测试用例检测结果独立记录
- 便于对比不同模型表现

### 4. 统计 API 修复
- 确保 `/api/stats` 从真实数据统计
- 确保 `/api/history` 正确查询记录

## 数据字段映射

| 数据库字段 | 来源 | 说明 |
|-----------|------|------|
| `userPrompt` | 用户输入 | 原始输入文本 |
| `mockModelOutput` | 模型原始输出 | 脱敏/改写前 |
| `finalResponse` | 处理后输出 | 脱敏/改写后 |
| `inputAction` | 输入检测结果 | allow/block/warn/mask/rewrite |
| `inputScore` | 输入风险分数 | 0-100 |
| `inputSummary` | 输入检测摘要 | 风险说明 |
| `outputAction` | 输出检测结果 | 同上 |
| `outputScore` | 输出风险分数 | 0-100 |
| `finalAction` | 最终决策动作 | 最严格的动作 |
| `targetProviderId` | 目标模型ID | 用户选择的模型 |
| `policyId` | 策略ID | 使用的检测策略 |
| `durationMs` | 总延迟 | 检测耗时 |

## 是否有原型设计

是（设计引导工具已开启，但本次主要涉及后端数据流修复，前端页面变化较小）

## 实施步骤

1. **阶段一：原型设计** — 加载 design-canvas 技能，设计检测会话记录的数据结构和交互流程确认（页面布局无大变化，主要是数据流调整）
2. **创建检测记录服务** — 新建 `src/lib/detection/recorder.ts`，实现 `recordDetectionSession()` 方法
3. **修改检测工作台** — 修改 `src/app/page.tsx`，在 sendMessage 完成后调用记录服务
4. **修改多模型评测** — 修改 `src/app/model-eval/page.tsx`，在评测完成后调用记录服务
5. **验证数据流** — 运行测试，确认检测记录正确写入，看板和历史页面展示真实数据
